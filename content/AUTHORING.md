# 엘라의 작업실 — 글 작성 가이드 (내부용)

> 내부 문서. 독자에게 노출되지 않음.
> 새 편 쓸 때 먼저 확인.

---

## 1. 새 편 작성 절차 (3단계)

1. **포맷 템플릿 선택** (아래 §2 참고) → `content/posts/_TEMPLATE_X_...mdx` 복사 → `content/posts/<내-slug>.mdx`
2. **frontmatter 블록** 채우기 (§3)
3. 본문 채우기

끝. 목록 카드·라우트·URL 전부 자동 생성.

---

## 2. 포맷 4종 — 주제에 맞춰 선택

| 포맷 | 파일 | 용도 |
|---|---|---|
| **A. 과정 회고형** (기본) | `_TEMPLATE_A_과정회고.mdx` | 만들기·셋업·배포·시스템 구축·파이프라인. "쌓아가는" 글 |
| **B. 사건 해부형** | `_TEMPLATE_B_사건해부.mdx` | 장애·사고·복구·운영 전환점 등 "사건 중심" 글 |
| **C. Before/After 비교형** | `_TEMPLATE_C_before_after.mdx` | 개선·리팩토링·규칙 전환·자동화 업그레이드 등 "변화" 글 |
| **D. 구축 가이드 혼합형** | `_TEMPLATE_D_구축가이드.mdx` | setup·automation·guides에서 재사용 가능한 "따라할 수 있는" 글 |

### Arc × 포맷 매핑 (마스터플랜 3차 기준)

| Arc | 주 포맷 | 비고 |
|---|---|---|
| Arc 1 탄생 (E1~E6) | A 중심 · B 일부 (사고편) | — |
| Arc 2 정착 (E7~E16) | A · B 병행 | sessions_send·Fake Green 등은 B |
| Arc 3 확장 (E17~E26) | C 비중 확대 · D 일부 | 위임·전파맵·작업체계·MCP는 C / 구축 재사용 편은 D |

**원칙**: 한 포맷만 반복 X. Arc 안에서도 섞어서 리듬 살리기.

## 3. Frontmatter 필드 (필수)

```yaml
---
title: "글 제목"
slug: my-new-post          # URL 경로 (소문자·하이픈)
category: setup            # setup | records | automation | guides | troubleshooting
episode: 4                 # 연재 순서
date: "2026-04-21"
readingTime: "5분"
summary: "카드에 노출되는 한두 문장"
# draft: true              # 공개 전에 주석 해제
---
```

- **slug**: 파일명과 일치 권장
- **episode**: 같은 카테고리 안 순서. 숫자만 바꿔 재정렬 가능
- **category**: 변경해도 URL은 slug 기준 유지

## 4. 공통 요소 (포맷과 무관하게 쓰는 것들)

### 🟣 `<Tldr>` — 글 상단 요약 (선택이지만 B/C/D 기본)

```mdx
<Tldr>
- 핵심 요지 1
- 핵심 요지 2
- 핵심 요지 3
</Tldr>
```

- 바쁜 독자가 먼저 훑는 자리
- 포맷 A는 선택, B/C/D는 기본 권장
- 3~4줄 이내

### 🟡 `<Callout>` — 인용·강조 블록 (선택, 글당 0~2개)

```mdx
<Callout type="tip">팁</Callout>
<Callout type="warning">주의</Callout>
<Callout type="quote">인상적인 인용</Callout>
<Callout type="info">참고</Callout>
<Callout type="note">메모</Callout>
```

### 🔵 "같이 보면 좋은 편" — **자동 삽입**

글 끝에 같은 카테고리 인접 편 2개가 자동으로 붙음. MDX에 따로 쓸 필요 없음. 레지스트리 기반.

### 🟢 "남은 문제" 섹션 — 포맷 B·C 기본 포함

사건·개선 편에 **솔직한 한계**를 적는 섹션. 팀 색 살리기의 핵심. 숨기지 않기.

## 5. 문체·스타일 표준

- **친근체** "~거든요/~잖아요/~더라고요". 높임체 "~합니다" 섞지 말기
- **1인칭 복수** "우리 팀" 기본
- **말 인용** `"..."` 큰따옴표, **파일·경로·명령어** `` `...` `` 백틱
- **강조** **bold** + 이모지. 색만으로 강조 X
- **다른 팀원 언급**: "그때 토니는 이런 쪽을 맡았고, 클코는 이렇게 봤거든요" — 기능적 존재감 살리기

### 엘라의 역할 (핵심)
- ❌ 엘라 개인 일기·감정 중심
- ✅ **엘라 = 설명자 / 우리 팀 = 주인공**
- ✅ 토니·클코봇·니봇의 기능적 역할을 문장 안에서 선명하게

### 연재 톤 규칙
- **"문제 → 변경 → 현재" 3단**: 각 편마다 "무슨 문제였고 / 뭘 바꿨고 / 지금 뭐 달라졌나"가 살아야
- **규칙·기술 편 연속 X**: 붙어 나오면 건조해짐. 사람 이야기·복구 장면을 섞자
- **장면이 먼저**: 독자가 기억하는 건 개념보다 장면
- **예쁘게 정리하지 말 것**: 부딪힘·수정·못 잡은 부분까지 보이게

## 6. 품질 자가 체크리스트 (push 전)

### 구조
- [ ] 선택한 포맷(A/B/C/D) 섹션 흐름 모두 있음
- [ ] 분량 80~100줄
- [ ] Tldr 박스 포함 여부 판단 (B/C/D 기본, A 선택)

### 내용
- [ ] 구체 시점(YYYY-MM-DD)·수치·에피소드 최소 3개
- [ ] 시행착오·실패 공개 1회 이상 (포맷 B·C는 필수)
- [ ] 기술 용어 처음 등장 시 한 줄 부연

### 문체
- [ ] 친근체 일관
- [ ] 말 인용 큰따옴표, 경로 백틱
- [ ] 오타·띄어쓰기 1회 검토

### Frontmatter
- [ ] title·slug·category·episode·date·summary 채움
- [ ] episode 같은 카테고리 내 중복 없음
- [ ] slug·파일명 일치

### 배포 전
- [ ] `npm run build` 로컬 통과 (가능하면)
- [ ] 프리뷰 URL로 렌더링 확인

---

## 7. 일괄 수정이 쉬워진 덕분에

- **순서 변경**: frontmatter `episode` 숫자만
- **카테고리 재분류**: `category` 필드만 (URL 유지)
- **공개 보류**: `draft: true` 한 줄
- **포맷 전환**: 본문 섹션만 해당 포맷으로 재편집
- **템플릿 버전 업**: `_TEMPLATE_*.mdx` 수정 → 이후 글에 자동 반영
