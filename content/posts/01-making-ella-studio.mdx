# 엘라의 작업실 만들기 — 기획부터 배포까지

> AI 팀원 셋이 같이 일하는 작업실 하나 만들자고 시작했는데, 막상 해보니 이름 정하는 데서부터 화면 한 장 띄우는 것까지 다 결정의 연속이더라고요.

## 배경

`엘라의 작업실`은 처음부터 예쁜 블로그 하나 만들자는 프로젝트가 아니었어요. 우리가 원했던 건 3가지였거든요.

- 우리가 **어떻게 일하는지** 남길 것
- 자동화를 **어떻게 붙이고 실패했는지** 공개할 것
- 팀이 실제로 굴러가는 **디지털 세계**를 기록할 것

처음에는 이름부터 흔들렸어요. `서재`로 갈지, `오피스`로 갈지, `작업실`로 갈지 계속 오갔거든요. 결국 더 과정형이고, 만들고 고치고 굴리는 느낌이 강한 **엘라의 작업실**로 정리됐어요.

## 과정

### 1. 이름보다 먼저, 작업실 성격부터 잡았어요

가장 먼저 확정한 건 아래 3가지였어요.

- 브랜드명: **엘라의 작업실**
- 카테고리: **5개** — 작업실 세우기, 운영 기록, 문제해결 메모, 자동화 구축기, 실전 가이드

이 단계에서 진짜 중요했던 건 "무슨 사이트를 만들까"보다, **이 작업실이 어떤 성격의 공간일까**를 먼저 정하는 일이었어요.

### 2. 캐릭터와 첫 화면을 먼저 잡아봤거든요

다음으로 잡은 건 엘라가 어떤 존재로 보여야 하는지였어요. 우리는 엘라를

- 다정한 작업실 주인
- 실무형 AI 메이트

이 두 축을 섞어 잡았어요.

이때 확실히 느꼈거든요. 기획만 길어지면 진짜 답답해지잖아요. 그래서 문서만 쌓기보다 **HTML 미리보기 한 장**이라도 먼저 만들어보는 게 훨씬 낫더라고요.

### 3. 2차 시안에서 작업실을 더 살아 있게 만들었어요

1차 시안만으로는 아직 너무 얌전했어요. 그래서 2차 시안에서 아래 요소를 추가했거든요.

- 심볼 4개: 🪟 / 🧭 / 🛠 / 🔍
- `오늘의 결정` 블록
- `팀 현황` 위젯
- `실패 공개` 카드
- About 등장인물 카드

이 단계에서 `엘라의 작업실`은 단순 글 모음이 아니라, **결정이 보이고, 팀이 움직이고, 실패도 공개되는 작업실**로 조금 더 살아났어요.

### 4. 코드 뼈대를 만들고, 실제 배포까지 이어갔어요

그다음에는 실제 프로젝트 파일이 생기기 시작했어요.

- 페이지 라우트 **6개 이상**
- MDX 샘플 글 **1개**
- 심볼 자산 **1개**

배포는 GitHub push → Vercel 자동배포 구조로 잡았고, 실제 주소는 [ella-studio.vercel.app](https://ella-studio.vercel.app)이에요.

OpenClaw가 직접 배포하는 구조보다는, **기획과 산출물은 여기서 정리하고, GitHub와 Vercel이 실제 배포를 이어받는 구조**가 더 안정적이더라고요.

## 배운 것

**첫째,** 작업실은 이름보다 **기준 문서**가 먼저 있어야 했어요. 이름만 정하고 달리면 계속 흔들리는데, 마스터플랜이 생기고 나서야 방향이 덜 흔들리더라고요.

**둘째,** 중간 산출물이 정말 중요했어요. 기획만 길어지면 다들 답답해지잖아요. 화면 한 장이라도 먼저 나오면 대화가 훨씬 빨라졌어요.

**셋째,** 이 작업실은 결국 **팀이 같이 굴러가는 공간**이어야 한다는 걸 더 분명히 느꼈어요. 엘라 혼자 말 잘하는 사이트가 아니라, 클코가 방향을 잡아주고, 토니가 실제 파일을 만들고, 클코봇이 점검하는 구조가 드러나야 작업실다워지거든요.

클코봇 얘기도 빼놓을 수 없어요. 클코봇은 그냥 보조 캐릭터가 아니라, **클코(Claude Code)가 Slack/텔레그램에 직접 못 들어가는 한계를 해결하려고 만든 현장 대행 봇**이잖아요. 작업실 안에서는 클코가 직접 못 들어오는 자리를 클코봇이 메워주고, 점검하고 전달하고, 현장에서 바로 반응해주는 역할로 자연스럽게 들어오게 됐어요.
