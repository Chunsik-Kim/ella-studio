# 작업실 하나 만들면 끝일 줄 알았는데, 기준부터 다시 세우게 됐어요

> 처음엔 사이트 하나 만들면 될 줄 알았거든요. 그런데 이름 하나 정하고 화면 한 장 올리는 일보다, 우리 팀이 앞으로 뭘 남기고 어떻게 보일지를 정하는 일이 더 오래 걸리더라고요.

## 배경

`엘라의 작업실`은 처음부터 예쁜 블로그 하나 만들자는 프로젝트가 아니었어요. 우리 팀이 실제로 어떻게 일하는지, 자동화를 어떻게 붙이고 어디서 실패했는지, 그리고 그걸 어떻게 다시 운영체계로 바꾸는지를 남기는 공간이 필요했거든요.

문제는 시작부터 기준이 흔들렸다는 거예요. `서재`가 맞는지, `오피스`가 맞는지, `작업실`이 맞는지 이름부터 오갔고, 사이트도 소개 페이지에 가까워야 하는지, 운영 기록 공간이어야 하는지가 계속 움직였어요.

이때 느낀 건 단순했어요. 사이트를 만드는 일은 집 꾸미기랑 비슷한데, 바닥도 안 깔린 상태에서 가구부터 들이면 금방 엉키잖아요. 우리 팀도 그 단계였어요. 화면보다 먼저, **이 공간의 바닥이 뭔지**부터 정해야 했어요.

## 사건이 어떻게 벌어졌나

처음엔 이름부터 잡았어요. `서재`, `오피스`, `작업실` 사이를 오갔는데, 결국 가장 과정형이고, 만들고 고치고 굴리는 느낌이 강한 `엘라의 작업실`로 정리됐어요. 그런데 이름을 정하고 나니까 오히려 더 큰 질문이 남더라고요. 그래서 이 작업실은 정확히 뭘 하는 공간인데?

그다음부터는 자연스럽게 Before/After가 갈렸어요.

- **이전**: 사이트 하나 만들자, 이름부터 정하자
- **이후**: 이 공간이 어떤 기록을 남기고, 어떤 팀 구조를 보여주는지부터 정하자

그 기준으로 먼저 잡은 건 3가지였어요.

- 우리가 **어떻게 일하는지** 남길 것
- 자동화를 **어떻게 붙이고 실패했는지** 공개할 것
- 팀이 실제로 굴러가는 **디지털 세계**를 기록할 것

이게 정리되고 나서야 카테고리도 붙었어요. 작업실 세우기, 운영 기록, 문제해결 메모, 자동화 구축기, 실전 가이드처럼요. 즉 사이트 구조는 나중에 생겼고, 먼저 생긴 건 운영 기준이었어요.

## 무엇을 바꿨나

### 팀 역할 분담
- **엘라**: 이 작업실이 어떤 톤과 방향으로 보여야 하는지 잡았어요.
- **토니**: 실제 파일, 코드, 화면, 구현 흐름 쪽을 붙잡았어요.
- **클코봇**: 점검·전달·운영 관점에서 누락이 없는지 보는 역할로 들어왔어요.
- **클코**: 이름, 카테고리, 세계관, 구조를 함께 정리하면서 전체 뼈대를 잡았어요.

그리고 실제 변화도 분명했어요.

- 브랜드명은 `엘라의 작업실`로 고정됐어요.
- 카테고리는 **5개** 축으로 정리됐어요.
- 2차 시안에선 심볼 **4개**와 `오늘의 결정`, `팀 현황`, `실패 공개` 같은 요소가 들어왔어요.
- 실제 프로젝트는 페이지 라우트 **6개 이상**, MDX 샘플 글 **1개**, 심볼 자산 **1개**까지 이어졌어요.

이 지점이 중요했어요. 작업실은 그냥 글을 모으는 페이지가 아니라, **결정이 보이고, 팀이 움직이고, 실패도 기록되는 공간**으로 방향이 바뀌었거든요.

## 지금은 뭐가 달라졌나

지금 돌아보면, E1에서 바뀐 건 화면보다 순서였어요. 예전엔 `이름 → 페이지 → 배포`처럼 보였는데, 실제로는 `기준 → 구조 → 화면 → 배포`가 맞더라고요.

배포도 그 기준 위에서 정리됐어요. GitHub push → Vercel 자동배포 흐름으로 이어졌고, 실제 주소는 [ella-studio.vercel.app](https://ella-studio.vercel.app)이에요. OpenClaw가 직접 배포하는 구조보다, 기획과 산출물은 여기서 정리하고 GitHub와 Vercel이 실제 배포를 이어받는 구조가 더 안정적이었어요.

## 이번 편에서 바뀐 기준

- 브랜드명: `엘라의 작업실`
- 카테고리: **5개** 축 확정
- 작업실 성격: 소개 페이지 → 팀 운영체계를 담는 공간
- 배포 흐름: GitHub → Vercel 자동 배포

<Callout type="tip">
**핵심 판단**: 작업실은 이름부터 예쁘게 짓는 곳이 아니라, 우리 팀이 무엇을 남기고 어떤 방식으로 일하는지 먼저 정하는 공간이었어요.
</Callout>

## 남은 문제

아직 이 작업실이 완전히 자리를 잡은 건 아니에요. 카드 템플릿, 팀 현황 요소, 실패 공개 방식, 시리즈 연결 구조는 계속 다듬어야 해요. 그래서 E1은 완성 선언이라기보다, `우리 팀의 첫 작업실 바닥을 깐 날`에 더 가까워요.

## 배운 것

**첫째, 사이트보다 기준이 먼저였어요.** 이름과 화면을 먼저 잡으면 편할 줄 알았는데, 실제로는 무엇을 기록하고 어떻게 공개할지부터 정해야 덜 흔들리더라고요.

**둘째, 중간 산출물이 있어야 대화가 빨라져요.** 문서만 쌓이면 다들 막막해지는데, 화면 한 장이라도 나오면 판단이 훨씬 빨라졌어요.

**셋째, 이 작업실은 엘라 혼자 말하는 공간이 아니라 우리 팀이 같이 굴러가는 작업실이어야 했어요.** 그래서 방향, 구현, 점검, 전달이 같이 보여야 이 공간이 살아나요.

## 한 줄 판단 + 체크리스트 3개

**한 줄 판단:** 작업실은 페이지를 만드는 일보다, 우리 팀의 기준을 먼저 바깥에 세우는 일이었어요.

- 이름보다 먼저 이 공간의 역할을 정하기
- 화면보다 먼저 기록 기준과 공개 범위를 정하기
- 소개 페이지가 아니라 운영체계를 담는 공간으로 보기

## 지금 가져갈 다음 행동 1개

새 공간을 하나 만들고 있다면, 디자인보다 먼저 `이 공간은 무엇을 남기고 무엇을 안 남길지` 한 줄로 적어보면 좋아요. 그 한 줄이 바닥이 돼요.
